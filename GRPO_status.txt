  The AOAE Policy in Detail                                                              
                                                                                         
  What the Policy Outputs (4 heads, not 1)                                               
                                                                                         
  The policy outputs 4 independent Bernoulli decisions per token position at each      
  diffusion step:
  ────────────────────────────────────────
  Head: head_unmask
  Symbol: u_t
  Valid on: masked [M] positions
  Meaning: Should this [M] be revealed to a predicted token?
  ────────────────────────────────────────
  Head: head_remask
  Symbol: r_t
  Valid on: unmasked positions
  Meaning: Should this previously-revealed token be reverted back to [M]?
  ────────────────────────────────────────
  Head: head_cache
  Symbol: κ_t
  Valid on: all positions
  Meaning: Should this position's KV state be committed to the dKV-Cache?
  ────────────────────────────────────────
  Head: head_access
  Symbol: q_t
  Valid on: all positions
  Meaning: Will this position be accessed in the next H steps? (auxiliary prediction)

  Constraint enforced at sampling time: κ_t * r_t = 0 — you can't simultaneously cache
  and remask a position.

  ---
  How Speculative Caching Actually Works

  The kappa_t head directly controls a DKVCacheManager. The inference loop has three
  phases per step (aoae/inference.py:266):

  1. Phase 1 (Remask): For positions where r_t=1, revert to [M] and invalidate those
  positions in the KV cache (cache_mgr.invalidate(r_t))
  2. Phase 2 (Unmask): For positions where u_t=1, run the dLLM forward pass to predict a
  token
  3. Phase 3 (Cache commit): For positions where κ_t=1, commit those positions to the KV
  cache (cache_mgr.commit(kappa_t))

  So the policy is explicitly learning: "after I decide a token here, is this position
  stable enough to cache its KV state?" A position that gets cached and then later
  remasked causes a thrash (a cache invalidation that wasted the prior cache commit). The
   reward penalizes this via -beta * sum(thrash_counts).

  The cache benefit in actual inference: positions with committed KV states don't need to
   be recomputed in the next forward pass — only positions that changed need fresh KV
  computation.

  ---
  Inputs to the Policy

  The policy gets a rich state per position (concatenated and projected):

  - H_t [D-dim]: The "soft-masked embedding" — a learned blend between the [M] embedding
  and the top-K predicted token embeddings. The blend weight λ is gated by token entropy
  via SoftMaskedState (the 3 learnable scalars ω_s, ω_a, ω_b control this sigmoid gate).
  High-entropy (uncertain) positions → closer to [M] embedding. Low-entropy → closer to
  the predicted token embedding.
  - m_t [1]: Binary mask indicator
  - q_feat [1]: PRISM quality score (a separately-trained model scoring output quality)
  - t/T [1]: Step fraction (how far through diffusion)
  - agreement [1]: Whether auxiliary and primary model agree on this position's token
  - age_feature [1]: How many steps since this position last changed (stable positions
  have high age)
  - last_action_feature [1]: Was this position accessed in the previous step?

  The bidirectional transformer backbone lets each position attend to all others — so the
   cache decision for one position can be informed by the global state of the sequence.

  ---
  What's Frozen vs. Trained

  Frozen (no gradients): The base dLLM (LLaDA2.1-mini) entirely — rollouts run under
  @torch.no_grad().

  Trained by GRPO:
  - AOAEPolicy (~31K params) — the 4-head transformer
  - SoftMaskedState — 3 scalar gating parameters (ω_s, ω_a, ω_b)
  - PRISMAdapter — the quality score head (pre-trained separately, loaded from
  prism_adapter.pt)

our Point on Speed Bonus: Diffusion Steps vs. Actual Time

  You're right to flag this. The current reward:

  speed_bonus = (T_hat / T)^alpha

  ...rewards fewer diffusion steps, which is only one axis of speedup. The KV cache's
  actual benefit is within-step wall-clock savings — if 80% of positions are cached, the
  forward pass for step t is much cheaper even if the total step count doesn't change.

  The current formulation captures step-count savings (do we need fewer passes?) but not
  within-step caching efficiency (is each pass cheaper?). The thrashing penalty
  indirectly rewards stable caching, but doesn't positively reward high cache-hit rates.

  The gap: if the policy learns to cache many stable positions but still runs the same
  number of diffusion steps, the speed bonus doesn't improve. It would take a
  hardware-measured latency signal (or a surrogate like cached_fraction * step_cost) to
  capture this.
